## 1. [HIGH] exportPad() discards the WAV writer's success/failure result and always reports success, so a failed export (disk full, denied write) is reported to the user as a completed export
**Where:** `Source/PluginProcessor.cpp:2909`  **Subsystem:** file-io

**Failure scenario:** exportPad() writes a rendered pad slice to WAV via writer->writeFromAudioSampleBuffer(buf, 0, buf.getNumSamples()) at line 2909, then unconditionally `return true;` at line 2910 without checking the call's bool return value. If the disk fills up mid-write, or a permissions/AV-lock issue truncates the write, writeFromAudioSampleBuffer returns false but exportPad still reports success. Both call sites treat this as authoritative: exportKit() (line 2968) sets `any = true` from it and reports overall export success in its return value, and the pad-drag-out chip (PluginEditor.cpp:269-274, `return p.exportPad (pad, f) ? f : juce::File();`) hands the truncated/corrupt WAV straight to the DAW's drag-drop as if it were valid. The user ends up with a silently truncated or empty WAV believed to be a good export -- compare to the sibling encodeFlacTo() in the same file (line 3112-3113), which correctly checks this exact same writer call's return value and propagates failure.

**Verifier's confirmation:** Verified directly in Source/PluginProcessor.cpp lines 2890-2911. exportPad() calls writer->writeFromAudioSampleBuffer(buf, 0, buf.getNumSamples()) at line 2909 and discards its bool return value, then unconditionally executes `return true;` at line 2910. There is no guard, no later check, and no exception path that would catch a failed write — JUCE's AudioFormatWriter::writeFromAudioSampleBuffer returns false on failure (e.g. disk full, stream write error) rather than throwing.

Concrete failure scenario: user drags a pad out to the DAW (PluginEditor.cpp:269-274) on a full/near-full disk, or a permissions/AV-lock issue interrupts the write. FileOutputStream successfully opens (line 2899-2901 check passes) and the writer is successfully created (line 2903-2906 check passes), but the actual sample data write fails partway through. exportPad still returns true, so the editor's `p.exportPad (pad, f) ? f : juce::File()` hands the truncated/corrupt file straight back as a valid drag-drop payload — the DAW receives a broken WAV believing it succeeded. Same discard happens in exportKit() (line 2968: `if (exportPad (i, f)) any = true;`), so a full-kit export can silently produce some truncated WAVs while reporting overall success.

Confirmed by direct contrast with the sibling encodeFlacTo() in the same file (lines 3099-3116), which performs the structurally identical writer->writeFromAudioSampleBuffer call in a loop and correctly checks its return value at line 3112, propagating an error string on failure. exportPad has no equivalent check — this is a straightforward omission, not an intentional design difference, since the two functions otherwise follow the same writer-lifecycle pattern (create output stream, create writer, release stream ownership, write, return status).

No guard elsewhere in the file rescues this: there's no post-write validation (e.g. re-reading the file, checking file size against expected sample count) anywhere in exportPad, exportKit, or the editor's drag-chip callback.

**Fix sketch:** Capture the bool from writer->writeFromAudioSampleBuffer(...) into a local (e.g. `const bool ok = writer->writeFromAudioSampleBuffer(buf, 0, buf.getNumSamples());`), and change the final line to `return ok;` — mirroring the existing pattern in encodeFlacTo() at line 3112-3113. No other changes needed since exportKit() and the editor's drag-chip already correctly branch on exportPad's boolean result.

## 2. [HIGH] handleNoteOn -> assignPadCue -> pushUndo() runs a heap-allocating, unsynchronized undo-history push on the audio thread
**Where:** `Source/PluginProcessor.cpp:2504`  **Subsystem:** rt-safety

**Failure scenario:** A MIDI note-on for an unassigned pad (notes 36-51 mapped to pads 0-15) arrives in processBlock's MIDI loop (PluginProcessor.cpp:3722-3729) and calls handleNoteOn(), which (PluginProcessor.cpp:2504-2520) calls assignPadCue(pad) when cues[pad] < 0. assignPadCue() (PluginProcessor.cpp:523-547) unconditionally calls pushUndo() (line 532) BEFORE doing anything else. pushUndo() (PluginProcessor.cpp:481-498) does history.erase(...)/history.push_back(snapshot()) twice on a plain std::vector<CueSnap> member (PluginProcessor.h:550) with no lock and no atomics, and undoPos (PluginProcessor.h:551) is a plain int. This is a genuine real-time violation (vector reallocation = heap alloc/free on the audio thread whenever history grows or needs shrinking to the 64-entry cap) AND a data race: the SAME history/undoPos are mutated from the message thread by pushUndo()/undo()/redo() on every UI edit (PluginEditor.cpp: undoBtn.onClick, drag onDragStart handlers, Ctrl+Z/Y keyPresses at PluginEditor.cpp:1331-1333, etc.) with zero synchronization between the two threads. The code's own comment at PluginProcessor.h:360-366 explicitly (and incorrectly) asserts pushUndo/undo/redo are 'Message thread only' and that assignPadCue is not among the callers verified by grep -- but assignPadCue's call to pushUndo() was missed by that audit. Concretely: user taps/MIDI-triggers an unassigned pad while simultaneously clicking Undo/Redo or dragging a grain knob (which also calls pushUndo) in the editor -> concurrent unguarded std::vector mutation from two threads (heap corruption / crash potential) and, independent of any editor interaction, an audio-thread allocation whenever the vector needs to grow/erase, which can cause an audible glitch or, on a memory-pressured system, a lock/wait inside the OS allocator on the audio thread.

**Verifier's confirmation:** Verified the full call chain directly in source: processBlock (Source/PluginProcessor.cpp:3664, the real JUCE audio-thread callback) iterates the incoming MidiBuffer synchronously and calls handleNoteOn() for every note-on (line 3726) with no queue/dispatch in between. handleNoteOn() (line 2504) checks `cues[pad] < 0` and calls assignPadCue(pad) directly (line 2516) for any unassigned pad hit by a MIDI note 36-51. assignPadCue() (line 523) unconditionally calls pushUndo() (line 532) before anything else, using the default snap=false (so the allocation-heavy grid/transient snap path in setCue() is correctly avoided on this path, but pushUndo() itself has no such guard). pushUndo() (lines ~481-498) performs `history.erase(...)` (redo-branch drop), `history.push_back(snapshot())` (vector growth → heap realloc), and a second `history.erase(history.begin())` once size>64 (full-vector shift of CueSnap elements, a nontrivial struct: 16 pads × (cue+end+mask+7 grain floats)) — all on a plain `std::vector<CueSnap> history` / plain `int undoPos` (PluginProcessor.h ~550-551) with zero lock/atomic protection.

Confirmed there are exactly two call sites for assignPadCue: PluginEditor.h:2063 (message thread, drag-drop assign, passes snap=true) and PluginProcessor.cpp:2516 (audio thread, via handleNoteOn, passes snap=false/default). The header's own contract comment (PluginProcessor.h ~355-366) claims pushUndo/undo/redo are "Message thread only" and that this was verified by grepping PluginEditor.cpp for ".undo()"/".redo()" call sites — but that grep never covered pushUndo()'s only non-editor caller (assignPadCue via handleNoteOn), so the documented invariant is actually false. Any MIDI note-on for an unassigned pad (e.g., from a hardware controller, host MIDI track, or DAW automation driving plugin MIDI input) — concurrently with a UI-thread undo/redo click, a grain-knob drag, or a drag-drop cue assignment (all of which also call pushUndo()) — causes a genuine unsynchronized concurrent mutation of the same std::vector/int from two threads, and independently, an unbounded audio-thread heap allocation/memmove whenever the vector grows or is trimmed to its 64-entry cap.

**Fix sketch:** Route the audio-thread trigger through the existing message-thread-safe path instead of calling pushUndo()/setCue() directly from handleNoteOn(): post the "assign cue for pad N at position P" request via the existing uiFifo/async-notify mechanism (the same pattern already used elsewhere in this file for message-thread dispatch) so pushUndo() and history mutation only ever execute on the message thread, exactly matching the documented (and now correctly enforced) contract.

## 3. [HIGH] Unsynchronized process-global statics in ensureOrtLoaded() raced by multiple plugin instances
**Where:** `Source/StemSeparator.cpp:60`  **Subsystem:** stem-engine

**Failure scenario:** GentSampler instances each run their own juce::Thread (PluginProcessor derives from juce::Thread) and each instance's doStemJob()/doTranscriptionJob() calls stemEngine.initialise()/transcriber.ensureLoaded(), both of which funnel into ensureOrtLoaded(). If a user drops GentSampler on two tracks in one FL Studio project (or runs the Standalone alongside FL Studio) and triggers separation/transcription on both before the first has finished loading, two worker threads call ensureOrtLoaded() concurrently. `static bool tried`, `g_ortReady`, `g_ortVersion`, `g_cudaAppend`, `g_dmlAppend`, `g_cudaDllsFound`, and `g_cudaSetDevice` are plain non-atomic statics with zero locking (no CriticalSection/std::mutex/std::call_once anywhere in the file). Both threads can see `tried==false` simultaneously, both proceed to LoadLibraryExW the same DLL and call `Ort::InitApi()` concurrently on the single process-wide `Ort::Global::api_` table, and both write `g_ortVersion`/`g_cudaAppend`/`g_dmlAppend` without synchronization -- a textbook data race (UB) that can produce a torn/partial API table read by one thread mid-write by the other, or duplicate LoadLibrary races. This is exactly the kind of static shared across plugin INSTANCES the audit asked to focus on, and it sits directly on the cold-init path that was just hardened against the CUDA-preload hang.

**Verifier's confirmation:** Verified by reading StemSeparator.cpp end-to-end and its callers. ensureOrtLoaded() (line 60) uses a bare `static bool tried` with no CriticalSection/std::mutex/std::call_once/atomic anywhere in the file (grep confirms zero hits), guarding the check-and-set `if (tried) return g_ortReady; tried = true;` plus writes to g_ortReady, g_ortVersion, g_cudaAppend, g_dmlAppend, g_cudaDllsFound, g_cudaSetDevice — all plain non-atomic file-scope statics, i.e. process-wide, not per-instance.

Threading model check confirms the scenario is not just plausible but trivially triggerable: GentSamplerAudioProcessor derives `private juce::Thread` (PluginProcessor.h:96), so every loaded plugin instance owns and starts its own worker thread running GentSamplerAudioProcessor::run() (PluginProcessor.cpp:910). run() unconditionally calls gentCheckStemEngine() (line 921) right at thread startup — before the wait-loop, with no user action required — and gentCheckStemEngine() calls ensureOrtLoaded() (StemSeparator.cpp:182). Separately, initialise() (called from doStemJob() via stemEngine.initialise()) also calls ensureOrtLoaded() directly at line 684, and Transcriber.cpp calls the public wrapper gentEnsureOrtLoaded() at line 77 for doTranscriptionJob(). None of these paths take any lock before calling ensureOrtLoaded(), and there is no higher-level singleton/lock serializing instance construction or thread startup across instances.

Concrete failure scenario: user loads GentSampler on two tracks in one FL Studio project (or Standalone + FL simultaneously). Both plugin instances construct and each starts its own juce::Thread; both threads independently reach run()'s startup gentCheckStemEngine() call within the same ~250ms window (or a user fires stem separation on both tracks back to back before either finishes). Thread A reads tried==false, sets tried=true, proceeds into LoadLibraryExW + GetProcAddress + Ort::InitApi. Before A finishes, Thread B also read tried==false (classic TOCTOU on a non-atomic bool with no memory barrier) and also proceeds — both now call Ort::InitApi() concurrently against the single process-wide Ort::Global::api_ table, and both concurrently write g_ortVersion/g_cudaAppend/g_dmlAppend/g_ortReady with no synchronization. This is undefined behavior: worst case a torn/partially-initialized API table is read by one thread while the other is still writing it, or duplicate concurrent LoadLibrary/InitApi calls corrupt shared global state consumed immediately afterward by makeSession()/session creation on both threads. This is a real, low-bar-to-trigger data race, not a contrived edge case — it fires on ordinary multi-instance usage without any exotic timing.

Minimal fix: wrap the entire body of ensureOrtLoaded() (the tried-check plus all the LoadLibrary/InitApi/GetProcAddress work and static writes) in a function-local static synchronization primitive — either `static std::once_flag once; std::call_once(once, [&]{ ... });` returning g_ortReady afterward, or a static juce::CriticalSection/std::mutex held across the whole check-init-set sequence. A JUCE-idiomatic alternative is a static juce::SpinLock guarding the tried/g_ortReady check-and-set exactly as done elsewhere in the codebase per the audit's own convention.

## 4. [HIGH] No cross-instance/cross-process lock around first-run model download to the shared Documents\GentSampler\models folder
**Where:** `Source/ModelDownloader.cpp:94`  **Subsystem:** stem-engine

**Failure scenario:** modelsDir is a single shared path (Documents\GentSampler\models) used by every GentSampler instance in every process on the machine, with no juce::InterProcessLock or any lock file guarding it (confirmed absent via search). If two instances (two tracks in one FL Studio project, each on its own worker thread per PluginProcessor : juce::Thread, or a Standalone run concurrently with FL Studio) both hit 'Separate'/'Transcribe' before the ~1.79 GB first-run download completes, both call ensureModelsPresent()->downloadOne() for the same missing file at the same time. Each opens its own juce::FileOutputStream on the identical <name>.part path and both write independently-fetched HTTP byte streams into the same file with no coordination, corrupting whichever byte ranges overlap. Worse, downloadOne()'s finalize step (finalF.deleteFile() followed by partF.moveFileTo(finalF)) is not exclusive either: instance A can finish, verify its SHA-256, delete+rename into finalF, and return success to its plugin instance (which proceeds to load/init the session) while instance B -- still mid-retry-loop on the same file because its own concurrently-corrupted .part failed checksum -- calls finalF.deleteFile() again on its next attempt, yanking the just-validated file out from under instance A, and then may fail to rewrite it (corrupted download), leaving the shared models folder in an inconsistent state and forcing a second ~1.79 GB re-download for one or both instances.

**Verifier's confirmation:** Verified by reading ModelDownloader.cpp in full plus its only call sites (doStemJob at PluginProcessor.cpp:1282 and doTranscriptionJob at :2715) and the header contract. There is genuinely no InterProcessLock, lock file, mutex, or any other cross-instance/cross-process coordination anywhere in ModelDownloader.cpp or its callers (grep for InterProcessLock across Source/ returns nothing). ensureModelsPresent/downloadOne operate purely on juce::File paths under the single shared Documents\GentSampler\models directory.

Concrete trigger: two PluginProcessor instances (e.g., GentSampler on two FL Studio tracks in one project, or a Standalone run alongside FL Studio) each run their own worker-thread loop (the wantStems.exchange(false)/doStemJob() pattern at PluginProcessor.cpp:958-978). If neither has the models yet and both trigger separation around the same time, both call GentModels::ensureModelsPresent(modelsDir,...) concurrently. Both independently open juce::FileOutputStream on the identical <name>.part path (ModelDownloader.cpp:147) with no shared-lock/exclusivity check beforehand, so their HTTP byte streams interleave into the same file — each instance's `done` byte counter and `out->setPosition` are purely local state, unaware of the other writer, so the resulting bytes are simply wrong/corrupted for at least one of them. The finalize step (finalF.deleteFile() then partF.moveFileTo(finalF), lines 203-208) is likewise unguarded: instance A can pass its checksum and rename into place, then instance B — still cycling its retry loop after its own corrupted .part failed the SHA-256 check at line 196 — deletes finalF again on its own attempt (either directly via its own finalize once it eventually gets a clean download, racing a delete against A's freshly-placed file, or indirectly by re-triggering a fresh downloadOne pass that calls partF.deleteFile()/recreates and eventually finalizes over A's file). This can force one or both instances to redo a ~1.79GB download and leaves a window where the shared file is missing or truncated while another instance (or a third one initialising stemEngine against the same modelsDir) may fail to load it.

This matches the architecture note in CLAUDE.md that audio-thread/worker-thread separation is per-instance and there is no documented singleton/cross-instance gate for model downloads; nothing in the surrounding code (retry loop, checksum check, resumable Range logic) accounts for or defends against a second writer to the same path.</reasoning>
<parameter name="fix_sketch">Add a juce::InterProcessLock (e.g. named "GentSampler_ModelDownloadLock") acquired for the whole ensureModelsPresent/ensureBasicPitchPresent duration (or at minimum around each downloadOne's write+finalize section), so only one process/instance downloads a given file at a time; other instances should block or poll-wait on the lock and then re-check modelsPresent()/fileComplete() before proceeding, since the file may already have arrived while they waited.

## 5. [HIGH] restoreLoadPath/wantRestoreLoad is stale-sticky across a state restore with no source path, causing a later restore to silently load the wrong file's audio under the new restore's cues
**Where:** `Source/PluginProcessor.cpp:3566`  **Subsystem:** threading

**Failure scenario:** Host calls setStateInformation() for project/state A that has a source (EXTRA/path=fileA.wav): applyStateTree() writes restoreLoadPath=fileA and wantRestoreLoad=true (PluginProcessor.cpp:3585-3592), and the worker has not yet woken (run()'s wait(250) window, PluginProcessor.cpp:930-932). Before it wakes, the host calls setStateInformation() again with a DIFFERENT state B that has no EXTRA/path (e.g. an empty/never-had-a-source project, or a preset saved before any file was loaded) -- applyStateTree()'s `if (path.isNotEmpty())` guard (line 3567) means the whole restoreLoadPath/wantRestoreLoad block is skipped entirely for B, leaving restoreLoadPath/wantRestoreLoad exactly as A left them, while state B's own cues/masks/bpm etc. ARE written unconditionally just below (lines 3596-3653). When the worker finally wakes, wantRestoreLoad.exchange(false) (line 950) is still true from A, so doRestoreLoadJob() (line 3408) reads restoreLoadPath==fileA and decodes+adopts fileA's audio (adoptSourceBuffer(..., keepCues=true), line 3467) underneath state B's already-restored cues -- state B ends up displaying/playing fileA's audio although its own saved project never referenced fileA. doRestoreLoadJob() also has no restoreGen check (unlike doAnalysisJob/doSectionApplyJob which snapshot genAtEntry at 1165/2012 and compare before applying), so it cannot detect that a newer, unrelated restore has landed since it captured the path.

**Verifier's confirmation:** Verified end-to-end in Source/PluginProcessor.cpp. applyStateTree() (line 3546) only ever writes restoreLoadPath/wantRestoreLoad inside the `if (path.isNotEmpty())` block (lines 3567-3593); grep confirms restoreLoadPath is written nowhere else and cleared nowhere. When a state restore's EXTRA has no "path" property, that whole block is skipped, leaving any previously-set restoreLoadPath/wantRestoreLoad untouched, while the same function unconditionally overwrites cues/cueEnds/padStemMask/bpm/key/etc. for the new state a few lines below (3596-3653) and returns immediately (wantRender=true; notify()). setStateInformation() (line 4241) calls applyStateTree() directly with no debounce/serialization, so two host calls in quick succession (state A with a source path, then state B with none) both run on the message thread before the worker thread's run() loop (which only wakes every wait(250)) processes the first wantRestoreLoad flag. doRestoreLoadJob() (line 3408) reads only restoreLoadPath (still fileA) and calls adoptSourceBuffer(..., keepCues=true) (line 3467), and adoptSourceBuffer's keepCues=true path (line 336) explicitly skips touching cues/cueEnds — so it swaps in fileA's decoded audio (`source = s`) underneath state B's already-restored cues, with no restoreGen check anywhere in doRestoreLoadJob (confirmed absent by reading the whole function), unlike doAnalysisJob/doSectionApplyJob which both snapshot and compare restoreGen before applying worker results. This is a real, concrete, reachable data-integrity bug (wrong audio silently loaded under a different project's cues), matching the codebase's own established pattern (DATA_INTEGRITY_SPEC.md, restoreGen) for guarding exactly this class of async-restore race — this exact mechanism is the kind of issue that pattern was introduced to prevent, but doRestoreLoadJob was missed.

**Fix sketch:** In applyStateTree(), snapshot restoreGen into doRestoreLoadJob the same way doAnalysisJob/doSectionApplyJob do: capture genAtEntry = restoreGen.load() when wantRestoreLoad is set (store it alongside restoreLoadPath under infoLock, e.g. a new restoreLoadGen field), and have doRestoreLoadJob() bail out (no adoptSourceBuffer call) if restoreGen.load() != that captured generation. Additionally, explicitly clear restoreLoadPath/wantRestoreLoad when path.isEmpty() (else branch or unconditionally before the `if (path.isNotEmpty())` check) so a no-source restore can never inherit a stale pending load from a prior state.

## 6. [HIGH] FileChooser completion lambdas capture the editor `this` with no liveness guard — use-after-free if the plugin window is closed while the dialog is open
**Where:** `Source/PluginEditor.cpp:336`  **Subsystem:** ui-wiring

**Failure scenario:** User clicks LOAD (or SAVE KIT / LOAD KIT / EXPORT KIT / Export selected Pad as WAV) which opens an async native file dialog, then closes/hides the plugin editor window (e.g. FL Studio lets you close the plugin GUI while a modal-ish OS dialog from a background thread is still up, or the host tears down the editor for any other reason while the dialog is open). JUCE's createEditor()/editorBeingDeleted() lifecycle (PluginProcessor.cpp:4256) lets the host destroy the editor independently of the FileChooser, which is not owned/joined by the editor's destructor. When the user then picks a file (or the dialog is dismissed), FileChooser::launchAsync's completion callback fires on the message thread and executes the captured `[this]` lambda, dereferencing the destroyed editor's `chooser` member and `this->p`, and calling processor methods (p.loadFile/p.requestKitSave/p.loadKit/p.exportKit/p.exportPad) through a dangling `this` — a use-after-free that can crash the host.

**Verifier's confirmation:** Verified directly in Source/PluginEditor.cpp. All five FileChooser call sites (loadBtn.onClick ~334-346, saveKitBtn.onClick ~351-364, loadKitBtn.onClick ~366-377, exportKitBtn.onClick ~379-390, and the export-pad chooser inside exportMenu's popup callback ~723-733) construct `chooser = std::make_unique<juce::FileChooser>(...)` as a plain editor member (declared `std::unique_ptr<juce::FileChooser> chooser;` at PluginEditor.h:2896) and call `chooser->launchAsync(..., [this](...) {...})`, capturing the raw editor `this` and dereferencing `p` (processor reference) and other editor members inside the callback body. The destructor `~GentSamplerAudioProcessorEditor()` (PluginEditor.cpp:839-842) only calls `setLookAndFeel(nullptr)` — it does not reset `chooser`, join/cancel anything, or set any liveness flag. `createEditor()` (PluginProcessor.cpp:4256) just does `new GentSamplerAudioProcessorEditor(*this)` with no special handling. A grep for `SafePointer` across Source/ returns zero matches, confirming no liveness guard is used anywhere for these callbacks.

Concrete failure path: user clicks LOAD, the native async file dialog opens (JUCE's FileChooser dialog runs independently of the editor's lifetime — it is not modal-owned by the Component, and its OS-level dialog/thread is not joined by anything in the editor). Host (e.g., FL Studio) closes/hides/destroys the plugin GUI window while that dialog is still open — a normal thing hosts do (closing the editor panel, switching plugin windows, or the DAW tearing down/recreating the editor for any reason) — which invokes `editorBeingDeleted`/destructor and frees the `GentSamplerAudioProcessorEditor` object, including the `chooser` member and all other editor state. The still-in-flight FileChooser (owned by its own async machinery, not by the freed `unique_ptr` once the owner is gone) later completes (user picks/cancels a file) and invokes the captured `[this]` lambda on the message thread, which dereferences the destroyed editor's `this` (accessing `p`/other members) and calls processor methods like `p.loadFile(f)` — a genuine use-after-free that can corrupt memory or crash the host process.

No mitigating guard exists: no SafePointer, no atomic "editor alive" flag checked in the lambda, no chooser cancellation on destruction, and JUCE does not automatically invalidate FileChooser callbacks when the owning object is destroyed.

**Fix sketch:** Capture a `juce::Component::SafePointer<GentSamplerAudioProcessorEditor> safeThis (this)` (or `juce::WeakReference`) instead of raw `this` in each of the five FileChooser completion lambdas, and check `if (safeThis == nullptr) return;` before touching `chooser`/`p`/other editor state; alternatively store `chooser` on the processor (which outlives the editor) or explicitly cancel/null the chooser in the editor destructor combined with a `SafePointer` guard.

## 7. [MED] padStemMask restored from saved state without the 0x3F bit-mask clamp that gent::stemMaskWithBit always applies
**Where:** `Source/PluginProcessor.cpp:3600`  **Subsystem:** engine-math

**Verifier's confirmation:** Verified against Source/PluginProcessor.cpp and PluginEditor.cpp. adoptSourceBuffer() (PluginProcessor.cpp:275-356) is called from two genuinely different threads with no mutual exclusion spanning the whole function:

2. Background worker thread: GentSamplerAudioProcessor privately inherits juce::Thread (PluginProcessor.h:96); its run() loop (PluginProcessor.cpp:930-954) polls every 250ms and, when wantRestoreLoad is set, calls doRestoreLoadJob() (line 3408), which itself calls adoptSourceBuffer() at line 3467.

applyStateTree() (called by the host on the message thread via setStateInformation) only validates the stored path, writes restoreLoadPath under infoLock, sets wantRestoreLoad=true, and returns immediately (PluginProcessor.cpp:3562-3593) — the actual decode happens later, asynchronously, on the worker thread. The decode itself is chunked with threadShouldExit() polls (~1 second per chunk, ADDENDUM T, lines 3449-3465), so for anything but a very short file this window is open for a non-trivial, user-observable amount of time (seconds).

Inside that window, nothing prevents the user from dragging a different file onto the UI. That triggers loadFile() -> adoptSourceBuffer() on the message thread while doRestoreLoadJob() -> adoptSourceBuffer() may simultaneously be executing on the worker thread. adoptSourceBuffer() updates state via three separate, independently-scoped locks (srcLock for `source` at 284-287, infoLock for fileName/filePath/transientOnsets/featureFrames at 288-296, stemLock for stemSet at 315-319) plus fully unguarded plain atomic writes to cues[]/cueEnds[]/detectedBpm/analysisKeepCues/wantAnalysis (336-350). Each individual lock scope is internally consistent, but nothing stops the two concurrent calls' scopes from interleaving with each other, so e.g. `source` can end up holding one call's buffer while `cues[]` (computed from `len`, that call's buffer length) was actually written by the other call — cue positions computed for one file's length get applied against a different file's audio.

I checked for any guard that would prevent this (restoreGen is the obvious candidate) — restoreGen is only used to invalidate stale analysis/classify results applied after a restore (checked at lines 1165/1186/1197/1213 and 2012/2048); it is never checked by loadFile() or adoptSourceBuffer(), so it does not serialize or block a concurrent direct load during an in-flight restore. There is no "restore in progress" flag, no lock spanning the whole adopt, and no thread-affinity check anywhere in the path. The race is real and reachable by ordinary user action (open a host project, then drag a new file onto the plugin UI before the restore's async decode finishes).

Severity is MED rather than HIGH: no crash is implied (each individual lock scope is memory-safe on its own), and the failure mode is silent, wrong-but-plausible state (mis-slicing / a stale filename display / wrong stem set) rather than a hang or data loss to disk. It's a real correctness bug but requires a fairly deliberate/unlucky user timing window to hit.

**Fix sketch:** Add a single mutex/CriticalSection (or a "loadInFlight" generation counter check like restoreGen) that wraps the entirety of adoptSourceBuffer() so only one caller can be inside it at a time; additionally, have loadFile() check/cancel a pending wantRestoreLoad (or bump restoreGen and have doRestoreLoadJob() check it before calling adoptSourceBuffer(), discarding its decode if a newer direct load has superseded it) so the two loads can't both land.
